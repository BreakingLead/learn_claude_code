# Agent Builder 挑战与解决方案

这份文档记录 Bee Agent 从 CLI/TUI 学习项目演进为可用 Agent Builder 过程中遇到的主要挑战，以及对应的解决方式。

## 1. 从硬编码运行时到模块化 Agent

**挑战：** 记忆、技能、todo、任务系统、mode、cron、多 Agent 等能力一开始很容易直接写进主循环，但这样会让 agent 越来越难理解，也很难给用户自定义。

**解决：** 运行时改为显式状态，并逐步抽象出模块化接口。能力模块可以贡献 system prompt、工具、hook、policy 或运行时服务；mode 则变成可配置对象，用来注入提示词并限制工具能力。

**结果：** 项目展示出更清晰的分层：agent loop 负责执行，模块负责能力定义，UI 负责可视化配置，持久化系统负责跨会话状态。

## 2. 区分 Agent Blueprint 和多 Agent Workflow

**挑战：** 单个 Agent 的定义图和多 Agent 编排图都可以用节点编辑器表示，但语义完全不同。混在一起会让 UI 和运行时含义变得模糊。

**解决：** 明确定义两个概念：

- **Agent Blueprint：** 描述一个可运行 Agent 的能力、提示词、工具、记忆和策略。
- **Agent Workflow：** 描述多个 Agent 之间的消息流、数据流和汇总关系。

UI 上也拆成 `Agents` 和 `Workflows` 两个顶层区域。

**结果：** 新用户的主路径更简单：先创建单个 Agent，连接能力，验证并试跑；之后再进入多 Agent 编排。

## 3. Node Editor 加载与前端打包

**挑战：** 早期 Web UI 出现过黑屏、只能看到 fallback DOM 等问题。根因是前端加载方式不正规，没有像真实应用一样打包和服务静态资源。

**解决：** 改为 Vite 正规打包模式，由 Go 后端服务构建后的静态资源；同时用生产构建和 headless browser 检查验证页面能正常加载。

**结果：** 节点编辑器从临时 demo 变成了可以稳定打开和演示的产品界面。

## 4. 类型化端口与非法图

**挑战：** 如果边是自由连接的，用户可以把工具输出连到 prompt 输入、把布尔值连到 memory 输入，最终生成无意义的图。Developer/Reviewer workflow 也暴露出数据依赖问题：Reviewer 应该依赖 Developer 的输出，而不是并发独立执行。

**解决：** 引入类型化端口和连接校验。端口类型包括 `prompt`、`toolset`、`memory`、`value`、`boolean`、`output` 等，并用类似 Blender 的颜色编码辅助识别。

**结果：** 图本身承载了更多语义，错误连接可以在交互阶段被拒绝，而不是等到运行时才失败。

## 5. Prompt 顺序与 Agent 聚合点

**挑战：** prompt 模块不一定满足交换律。角色提示词、policy 提示词、任务提示词的顺序可能影响模型行为。

**解决：** 把 Agent 节点设计成显式聚合点。能力节点连接到 Agent 节点，由 Agent 节点负责组装运行时能力；需要顺序时，可以用多个 prompt 输入端口表示稳定顺序。

**结果：** Agent 的组装过程从隐藏的全局状态变成了可见的图结构。

## 6. 可复用 Composite 节点

**挑战：** 用户需要复用常见能力组合，但如果做成实时共享的嵌套图，会很早引入同步、版本和调试复杂度。

**解决：** Composite 采用“打包子图为可复用节点”的方式。用户可以框选节点创建 composite，在左侧资产库中引用，也可以删除 composite。需要深度修改时，预期方式是手动复制后调整，而不是隐式修改所有引用。

**结果：** 项目支持复用，同时避免了过早引入难以解释的共享状态。

## 7. JSON 是底层格式，但不是主界面

**挑战：** JSON 对持久化、测试和调试很有价值，但如果把 JSON 编辑器放在主界面，会让产品像内部工具，而不是可视化编辑器。

**解决：** 视觉编辑器成为默认主界面；JSON 编辑器被收进底部可拖拽抽屉，只在高级检查和手动编辑时展开。

**结果：** 用户可以用图形界面完成主要工作，同时底层格式仍然透明、可版本化、可测试。

## 8. 数据流节点的内联默认值

**挑战：** 比较节点、选择节点这类数据流节点，如果每个输入都必须来自其他节点，小例子会很繁琐。Blender 一类节点编辑器通常允许未连接输入使用本地默认值。

**解决：** `value` 和 `boolean` 类型的输入端口在未连接时显示内联输入控件。例如 `Math Compare` 的 A/B 可以直接输入数字，`Select` 的 condition 未连接时可以直接选择布尔值。

**结果：** 小图可以快速搭建；复杂图仍然可以通过连接端口表达完整数据流。

## 9. TUI 转义字符问题

**挑战：** TUI 中拖动、滚轮和鼠标事件偶尔会把终端转义序列写进对话框，比如颜色查询或鼠标事件编码。

**解决：** 收紧 Bubble Tea 输入处理，区分鼠标事件、拖动状态和滚动 fallback；同时把复杂的节点编辑迁移到 Web UI，因为图编辑天然更适合浏览器指针模型。

**结果：** TUI 保留为对话和运行时观察界面，复杂可视化配置交给 Web Node Editor。

## 10. 配置与可复现演示

**挑战：** 只依赖环境变量会让演示不稳定，尤其是从不同工作目录启动时容易读不到配置。

**解决：** 运行配置逐步迁移到命令行选项和显式启动状态；本地运行产物如 `.agents/` 加入 `.gitignore`。

**结果：** 项目更容易演示、更容易复现，也更适合分享给别人。

## 11. 控制验证成本

**挑战：** 全量测试可能触发慢速或有成本的集成路径，不适合每次小改都跑。

**解决：** 对改动模块跑定向测试。例如节点编辑器改动主要跑 `go test ./internal/agent/node_editor ...`，前端改动跑 `pnpm run build`。

**结果：** 反馈速度更快，同时仍然覆盖当前改动的关键行为。

## 总结

这个阶段最大的设计变化，是从“一个不断增加硬编码能力的 agent”转向“由显式能力模块组装出来的 runtime”。节点编辑器把这个架构可视化了：类型化端口表达依赖，Agent 节点表达聚合，Composite 表达复用，验证和 dry-run 则把设计闭环到真实执行。
